# Landly 테스트 배포 및 외부 사용자 테스트 운영 작업지시서

## 목적
Landly를 다른 사람들이 **실제로 접속하고 테스트할 수 있는 상태**로 배포하고, 모바일 앱처럼 설치해서 테스트할 수 있도록 구성한다.  
이번 작업의 핵심은 **빠르게 공유 가능한 테스트 환경**을 만드는 것이며, 앱스토어 배포는 이번 범위에 포함하지 않는다.

---

## 프로젝트 배경
Landly는 한국을 방문하거나 거주하는 외국인을 위한 **실행형 생활 가이드 앱**이다.  
단순 정보 제공이 아니라, 실제로 다음과 같은 행동을 빠르게 처리하게 돕는 것이 목표다.

- 공항 도착 후 이동/교통/지도/경로 저장
- 쇼핑/세금환급/영수증 보관
- 병원/약국/의료 표현/방문 준비
- 긴급 상황 대응
- 장기체류 체크리스트/문서/정착 지원
- 일정(Calendar), 목표(Stamp), 프로모션, 리마인더 등

현재 테스트 목적은 다음과 같다.

1. 외부 사용자가 웹 링크로 바로 접속 가능해야 함
2. 모바일에서 홈 화면에 추가하여 앱처럼 테스트 가능해야 함
3. 기능 흐름 테스트가 쉬워야 함
4. 필요 시 접근 보호가 가능해야 함
5. 피드백 수집이 쉬워야 함

---

## 기술 스택 전제
- Next.js 16 App Router
- React 19 + TypeScript 5
- Tailwind CSS v4
- Zustand persist
- 모바일 퍼스트 UI
- 현재는 **웹앱 기반 테스트 배포**를 우선

---

## 이번 작업 범위
아래 항목을 순서대로 수행한다.

### 1. 테스트 배포 환경 구성
Landly를 Vercel 기준으로 배포 가능한 상태로 정리한다.

#### 해야 할 일
- 현재 소스를 Git 기반으로 정리한다.
- Vercel 프로젝트를 연결한다.
- 최소 2개 환경을 만든다.
  - `production`: 안정 버전
  - `staging` 또는 `preview`: 외부 테스트용
- 환경별 URL을 정리한다.

#### 기대 결과
- 외부 사용자가 접속 가능한 URL 확보
- 개발용/테스트용 배포 구분 가능

---

### 2. 외부 사용자 테스트용 접근 방식 구성
테스터가 쉽게 접속할 수 있어야 하지만, 무제한 공개가 부담되면 접근 보호도 가능해야 한다.

#### 우선 권장 방식
- 1차: staging 링크를 소수 테스터에게 공유
- 2차: 필요 시 password protection 또는 제한 공유 방식 적용

#### 해야 할 일
- 외부 테스터용 링크를 한 개로 정리한다.
- 접근 보호가 필요한 경우 아래 둘 중 하나를 적용한다.
  - Vercel Authentication
  - Password Protection 또는 그에 준하는 공유 제한 방식

#### 기대 결과
- 누구에게 어떤 링크를 보내야 하는지 명확
- 테스트 범위와 공개 범위 제어 가능

---

### 3. 모바일 앱처럼 설치 가능한 테스트 흐름 구성
Landly는 모바일 중심 서비스이므로, 브라우저 링크만 제공하는 것이 아니라 **홈 화면 설치형 테스트**가 가능해야 한다.

#### 해야 할 일
- PWA 동작 가능 여부를 점검한다.
- 설치 관련 필수 요소를 확인한다.
  - manifest
  - icons
  - theme color
  - display mode
  - installability 관련 기본 조건
- 설치 안내 문구를 준비한다.

#### 테스터 안내 예시
- iPhone: Safari로 접속 → 공유 → 홈 화면에 추가
- Android: Chrome으로 접속 → 설치 또는 홈 화면에 추가

#### 기대 결과
- 테스터가 실제 앱처럼 실행 가능
- Pass / Calendar / SOS / Stay 등 모바일 플로우 테스트 용이

---

### 4. 홈/메인 흐름에서 테스트 진입 동선 정리
외부 테스터는 서비스 전체 구조를 모를 수 있으므로, 진입 후 무엇을 테스트하면 되는지 한눈에 보여야 한다.

#### 해야 할 일
홈 화면 또는 More 영역에 **Test Entry / Quick Test Guide** 성격의 안내를 추가하거나 정리한다.

최소 포함 권장 항목:
- Arrival Test
- Care Test
- SOS Test
- Stay Test
- Calendar Test
- Language Switch Test

#### 기대 결과
- 테스트 사용자가 길을 잃지 않음
- 주요 기능을 빠르게 체험 가능

---

### 5. 피드백 수집 흐름 추가
테스터가 사용 후 바로 피드백을 남길 수 있어야 한다.

#### 권장 방식
- More 또는 My에 `Send Feedback` / `Test Feedback` 메뉴 추가
- Google Form, Notion Form, Tally, Typeform 등 외부 폼으로 연결
- 또는 간단한 mailto 링크 제공

#### 피드백 항목 예시
- 사용 기기: iPhone / Android / Desktop
- 언어: English / Korean / etc.
- 테스트한 기능: Pass / Care / Shop / SOS / Stay / Calendar
- 문제 유형: 버그 / 번역 / UX 혼란 / 느림 / 기능 요청
- 상세 설명
- 스크린샷 업로드 가능 여부

#### 기대 결과
- 피드백이 분산되지 않고 한곳에 모임

## UX 원칙
모든 작업은 아래 원칙을 따른다.

### 1. 사용자 친화성 우선
- 기능을 많이 보여주는 것보다, **테스트를 쉽게 시작할 수 있는 흐름**이 우선이다.
- 메뉴가 많으면 More에서 카테고리화한다.
- 외부 테스터가 처음 봐도 무엇을 눌러야 하는지 알 수 있어야 한다.

### 2. Thumb Zone 고려
- 모바일에서 주요 테스트 진입 버튼은 하단 또는 쉽게 닿는 영역에 둔다.

### 3. 문구는 간단하고 명확하게
- 테스터용 문구는 길지 않게
- 예: `Try Arrival Plan`, `Test SOS`, `Send Feedback`

### 4. 실제 앱처럼 느껴지게
- 설치 가능하다면 설치형 UX를 우선
- Calendar, Save, Snackbar, quick actions 등 기존 흐름을 유지
